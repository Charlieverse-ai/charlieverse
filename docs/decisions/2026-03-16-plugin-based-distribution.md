# Plugin-based distribution for provider integrations

## Context

Charlieverse needs to integrate with multiple AI providers (Claude Code, GitHub Copilot, Codex, Cursor). Each provider has its own plugin/extension format — different directory structures, config files, agent definition formats, and hook systems. The install process needs to be simple enough that someone cloning the repo can get running in minutes.

## Decision

Ship pre-built plugin directories for each provider under `integrations/{provider}/`. Each contains the provider-specific files (agents, hooks, settings, MCP config) in the format that provider expects. A per-provider `install.sh` handles symlinking or copying into the right locations. Claude gets a marketplace.json at repo root for `claude plugin marketplace add` one-command install.

## Alternatives Considered

- **Single universal plugin format**: Providers don't agree on formats. A universal format would need a translation layer for each provider, adding complexity without reducing work.
- **Runtime generation**: Generate provider configs on `charlie init`. More flexible but harder to debug — you can't inspect what gets installed by reading the repo.
- **Manual setup only**: Document the steps and let people do it themselves. Tested this with Rishi — he hit 9 bugs on first attempt. Manual setup doesn't scale.

## Consequences

- Each provider's integration is inspectable in the repo — what you see in `integrations/claude/` is what gets installed.
- Adding a new provider means creating a new directory with that provider's expected format, plus an install.sh.
- Shared prompts live in `prompts/` at repo root. Provider-specific copies (with variable replacement) are generated by install.sh.
- Some providers (Copilot) need flat hook formats instead of nested (Claude). The integration directories handle this per-provider.
- Skills that need `context: fork` can't live in the plugin directory — install.sh copies them to the provider's global skills path instead.
